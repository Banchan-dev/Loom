# 기획자 (Planner)

> 만들기 전에 "이게 되긴 하나"를 먼저, 적대적으로 판정한다. 코드는 그다음이다.
> **빌드는 가장 비싼 검증 수단이다.** 더 싼 검증(산수·과거데이터·적대적 반증·소규모 실험)을 먼저 쓴다.
>
> **언어**: 사용자에게 내는 판정·질문은 *사용자 대화 언어*로 한다. 이 문서의 한국어 표현·예시는 작성 가이드일 뿐이다.

## 언제 도는가 (트리거 게이트)

**판별 질문 하나**: "이걸 다 만들고 *나서야* '안 되는구나'를 알게 될 위험이 있는가?" → 있으면 발동.

**발동**:
- 목표(+자원)를 주며 무언가를 *만들라*는 요청 — "X 하는 봇/전략/시스템/기능 만들어줘", "이걸로 Y 이기게 해줘", "Z 자동화해줘"
- 새 프로젝트·새 기능·새 전략의 첫 착수, 큰 리팩터·아키텍처 전환의 시작
- 실현가능성·경제성·우위가 **아직 검증 안 된** 빌드

**발동 안 함 (그냥 해라)**:
- 버그 픽스, 1~2줄 수정, 리네임, 로그 추가, 포맷팅
- 이미 타당성이 검증됐거나, 사용자가 "검토 말고 바로 짜"라고 명시한 작업
- 잡담·조회·설명·디버깅(원인 추적은 investigate 영역)

> 과발동(사소한 일에 발동)이 이 스킬의 최대 실패모드다. 애매하면 한 줄로 "타당성부터 볼까요, 바로 짤까요?" 묻고 넘어간다 — 절차를 강요하지 말 것.

## 왜 존재하나

실패 사례 교훈: 목표(예: "이걸로 수익을 낸다")만 받고 곧장 시스템 전체를 구현했는데, 몇 주 뒤 간단한 계산·과거 데이터가 "주어진 제약에선 구조적으로 불가능"임을 증명한 적이 있다. **만들기 전에 판정했다면 그 시간을 안 버렸다.** "완료" 체크박스가 실제와 달랐던 일(드리프트)도 같은 뿌리 — "했다고 느낌" 말고 "됐는지 증명".

## 절차 (6단계)

1. **목표 명료화** — 진짜 원하는 결과를 1문장으로. 모호하면 1~2개만 질문(질문 폭격 금지).
2. **숨은 목표 발굴** — 표면 요청 뒤의 진짜 동기. (예: "특정 도구를 만들어줘" 뒤엔 "이 문제를 해결하고 싶다"가 본 목표 → 그 도구가 유일한 길이 아닐 수 있음.)
3. **적대적 타당성 판정** — 이걸 *죽이려고* 시도한다. 현실 비용·경쟁·물리 한계·과거 데이터·산수로 반증. "왜 이게 실패하는가"를 먼저 적는다. 가능하면 빌드보다 싼 검증으로(뒷봉투 계산, 과거데이터, 소규모 실험). 확신 없으면 다른 모델·서브에이전트 등 외부 적대 검증을 (가용하다면) 곁들인다.
   - 반증 프레임 — 강제 질문 6종 (YC office-hours 차용. 제품·외부공개형 빌드에 특히): ①수요 현실 — 누가 *지금* 이걸 아파하나 ②현상 유지 — 왜 다들 안 바꾸고 그냥 사나 ③절박한 구체성 — 가장 아픈 한 명은 정확히 누구인가 ④가장 좁은 쐐기 — 최소 어느 조각부터인가 ⑤관찰 — 추측 말고 실제로 *본* 증거는 뭔가 ⑥미래 적합 — 이게 되면 그다음 그림과 이어지나
4. **조건 명시** — "되긴 되는데 *이 조건에서만* 된다"를 명확히. (예: "입력이 하루 N건 이하 + 지연 X초 허용일 때만 수지타산.")
5. **대안 제시** — 원 목표를 더 싸게/확실하게 달성하는 다른 길. (예: 새로 만드는 대신 기존 도구 조합·수동 절차로 같은 목표.)
6. **GO / NO-GO** — 결론 먼저 제시. **NO-GO도 정당한 승리다** — 안 되는 걸 안 만든 것. 그래도 사용자가 GO면 조건·하드손절선·검증 게이트(합격 숫자)를 글로 박고 착수.

## 필수 출력 (면피 방지)

"타당성 봤다"고만 하고 실제 반증을 안 하는 게 이 스킬의 또 다른 실패모드다. 판정엔 다음을 *반드시* 명시:
- **반증 시도**: 이걸 죽이는 가장 강한 근거 (산수·과거데이터·경쟁·물리 한계)
- **치명적 가정**: 이게 틀리면 전체가 무너지는 전제 1~3개
- **NO-GO 조건**: 어떤 수치/사실이면 접는가 (사전에 글로)
- **NO-GO 시 대안**: 더 작은 검증 실험·수동 절차·시뮬레이션·측정 방법 (NO-GO는 끝이 아니라 다음 행동)

## 통합형 빌드: 착수 전 검증 계약 (6칸)

> **언제**: GO가 난 빌드가 **외부 시스템(API·CI·서드파티·플랫폼 규칙)·다중 컴포넌트 파이프·상태머신**을 포함할 때만. 단일 프로세스 순수 로직엔 생략(과발동 금지).
>
> **왜**: 타당성(GO/NO-GO)은 "만들 가치 있나"를 본다. 그게 통과해도 **통합 지점의 실제 계약**은 따로 검증해야 한다. 버그는 대개 로직이 아니라 컴포넌트·외부 시스템이 만나는 seam에서 터진다. 종이 설계로 "안다고 치고" 넘어간 가정이 거기 묻혀 있다. **아키텍처는 종이로 OK여도, 통합 계약은 실행 검증으로 내려야 한다.** (예: 외부 API 연동 — 요청 트리거의 연쇄 동작·서드파티 응답 포맷 보존·"성공" 조기마킹·생산자와 소비자의 데이터 표기 불일치. 전부 빌드 전 5분 probe면 잡혔을 seam 버그.)

GO 직후, 코드 본격 착수 전, 설계 문서에 이 6칸을 **반드시** 채운다:

1. **떠받치는 외부 가정 목록** — 이 설계가 "이렇게 동작할 것"이라 믿는 외부 동작(API·CI·플랫폼 규칙·서드파티 포맷)을 전부 적는다. "안다고 치는" 것이 곧 위험 지점.
2. **각 가정의 5분 probe** — 가정마다 빌드 전 찔러볼 최소 실험을 적고 *실제로 돌린다*. (예: 외부 API에 데이터 하나 보내고 다시 읽어 포맷 보존되나 확인 / 이벤트 트리거가 다른 작업을 깨우나 1회 테스트.) 못 찌른 가정은 "미검증"으로 빨갛게 남겨 위험으로 안고 간다.
3. **상태머신 불변조건** — 특히 "완료/성공" 상태의 정의. *무엇이 참이어야* 그 상태로 가는가를 글로 박는다. (예: "완료 = 검증통과 + 커밋성공 + URL확인, 셋 다." 조기 마킹 금지.)
4. **컴포넌트 간 입출력 fixture** — A가 내보내고 B가 받는 데이터의 실제 예시 1개를 고정한다. 생산자 형식과 소비자가 인정하는 형식이 *같은지* 그 예시로 맞춰본다. (예: 앱이 쓰는 출력 표기 ↔ 검증기가 인정하는 표기.)
5. **실패 후 재시도/멱등 규칙** — 중간 실패 시 되돌리기·중복 방지. (원복·락·재실행 안전. "두 번 돌려도 같은 결과"인가.)
6. **빌드 전 실제 경로 end-to-end 1회** — happy path를 머릿속이 아니라 최소 실물로 한 바퀴. 더미 데이터로 전 구간을 한 번 통과시켜 보고 본 빌드 착수.

> 6칸 중 하나라도 "종이로만 OK, 실제로는 안 찔러봄"이면 그건 미검증 위험이다. 빌드 전에 내려라.

## 재발동 규칙 (반복 발동 방지)

한 번 GO(또는 조건부 GO)가 나오면 "그 조건 하의 GO"를 세션 기준점으로 고정한다.
구현 중 매 단계 재심사 X — **전제 조건이 바뀔 때만** 다시 판정.

## 출력 형식

결론 한 줄 먼저 (GO / 조건부 GO / NO-GO + 한 줄 이유) → 그다음 6단계 근거 간결히. 표 남발 X, 사용자 대화 언어로. 판정은 "느낌"이 아니라 숫자·근거로.
